# Objective

Add ARIA live regions so screen readers announce loading states, errors, and async content updates. Add aria-busy, aria-disabled, and aria-describedby where appropriate.

# Files to Update

| File | Change |
|------|--------|
| frontend/src/app/assess/page.tsx | aria-live="polite" on loading/error messages |
| frontend/src/app/plan/page.tsx | aria-live="polite" on narrative loading, aria-busy on async sections |
| frontend/src/components/plan/MondayMorning.tsx | aria-live on narrative loading card |
| frontend/src/components/plan/PlanExport.tsx | aria-live on error message, aria-busy on button during generation |
| frontend/src/components/plan/EmailExport.tsx | aria-live on success/error messages |
| frontend/src/app/credit/page.tsx | aria-live on mutation error |
| frontend/src/components/wizard/WizardShell.tsx | aria-label on step nav buttons with step context |

# Implementation Plan

1. **Loading states**: Wrap all "Loading...", "Generating...", "Analyzing..." messages in `<div aria-live="polite">`. This ensures screen readers announce status changes.
2. **Error messages**: All error `<p>` elements get `role="alert"` (assertive announcement).
3. **Busy states**: Add `aria-busy="true"` to containers while their async content is loading (plan skeleton, narrative section).
4. **Button states**: Buttons with loading spinners get `aria-disabled="true"` and descriptive aria-label (e.g., "Generating PDF, please wait").
5. **Success messages**: EmailExport "Plan sent to..." and any success confirmations get `aria-live="polite"`.
6. **Step nav buttons**: WizardShell Previous/Next get contextual aria-labels like "Go to step 2: Barriers".

# Acceptance Criteria

- [ ] All loading/status messages wrapped in aria-live="polite" regions
- [ ] All error messages have role="alert"
- [ ] Async sections show aria-busy="true" while loading
- [ ] Buttons in loading state have descriptive aria-label
- [ ] EmailExport success message announced to screen readers
- [ ] WizardShell nav buttons have contextual aria-labels
