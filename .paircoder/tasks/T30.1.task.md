# Objective

Create a schedule-aware transit matching module that validates job shift times against M-Transit route schedules, detects transfer requirements, computes walk distances from stops to workplaces, and flags Sunday/night service gaps. Replace the current keyword-only transit detection in `job_matcher._filter_by_transit()` with route-data-driven feasibility checks. Wire enhanced transit info into the job scoring pipeline.

# Context

**Current state:**
- 14 M-Transit routes with `weekday_start`/`weekday_end`, `saturday`, `sunday` fields (all `sunday=false`)
- 43 stops with lat/lng and route_id references
- `_filter_by_transit()` in job_matcher.py uses keyword detection only (SUNDAY_KEYWORDS, checks text for "sunday", "weekend")
- `_is_near_transit()` does text matching on stop_name substring, NOT distance-based
- `TransitConnection` type exists but is never populated
- Rosa Parks Transfer Center is the hub stop (appears on 12 of 14 routes)
- `apply_transit_filter()` in filters.py exists but is NOT called in the main pipeline
- `job_scoring._score_transit()` returns 1.0/0.5/0.0 — no walk distance or transfer info

**Key data:**
- Route hours: ~5:00am–9:00pm weekdays, same Saturday, NO Sunday
- Stops: ~3 per route, coordinates available
- Rosa Parks Transfer Center: lat=32.3779, lng=-86.3088 (hub for transfers)

# Implementation Plan

1. **Types** — Add to `matching/types.py`:
   - `TransitWarning(str, Enum)`: `SUNDAY_GAP`, `NIGHT_GAP`, `LONG_WALK`, `TRANSFER_REQUIRED`
   - `RouteFeasibility`: route_number, route_name, nearest_stop, walk_miles, first_bus, last_bus, feasible (bool)
   - `TransitInfo`: serving_routes (list[RouteFeasibility]), transfer_count, warnings (list[TransitWarning]), google_maps_url (str|None)

2. **Schedule matcher** — New `transit_schedule.py`:
   - `find_serving_routes(lat, lng, stops, routes, max_walk_miles=1.0)` — find stops within walk distance, look up their routes, return RouteFeasibility list
   - `check_schedule_feasibility(shift_start_hour, shift_end_hour, route)` — compare against route weekday_start/weekday_end, return feasible + warnings
   - `detect_transfers(user_routes, job_routes)` — if user's nearest stops and job's nearest stops are on different routes, count transfers (via Rosa Parks hub)
   - `build_transit_info(job_lat, job_lng, shift_hours, stops, routes, user_lat=None, user_lng=None)` → TransitInfo
   - `google_maps_transit_url(origin_lat, origin_lng, dest_lat, dest_lng)` — generate directions URL with travelmode=transit

3. **Update job_matcher._filter_by_transit()**:
   - Fetch routes alongside stops (new query or join)
   - For each job with coordinates: call `build_transit_info()` instead of keyword matching
   - Populate `transit_accessible`, `route`, `sunday_flag` from TransitInfo
   - Fall back to keyword detection for jobs without coordinates

4. **Update job_scoring._score_transit()**:
   - Walk distance factor: <=0.25mi → 1.0, <=0.5mi → 0.9, <=1mi → 0.7, >1mi → 0.4
   - Transfer penalty: 0.8× for 1 transfer, 0.6× for 2+
   - Schedule feasibility: infeasible → 0.0, warnings → 0.7
   - Combined: `walk_score * transfer_mult * schedule_mult`

5. **Query enhancement** — Add to queries.py:
   - `get_routes_with_stops(session)` — join routes + stops for the full picture (or return both and let Python join)
