# Objective

Add a new step to the assessment wizard that collects household and benefits information. Step appears after Barriers and before Credit, and is fully skippable (defaults: household=1, income=0, programs=[]).

Backend stores the profile in the session and passes it through to the cliff calculator and PVS scorer.

# Implementation Plan

## 1. Backend Types (`types.py`)

Add `BenefitsFormData(BaseModel)` with: household_size (1-8), current_monthly_income (>=0), enrolled_programs (list[str]), dependents_under_6, dependents_6_to_17. Add `benefits_data: Optional[BenefitsFormData] = None` to `AssessmentRequest`.

## 2. Database (`database.py`)

Add `benefits_profile TEXT` column to sessions table DDL (JSON, nullable). Add to ALLOWED_COLUMNS.

## 3. Assessment Route (`assessment.py`)

Extract `benefits_data` from request, convert to `BenefitsProfile`, serialize as JSON, store in session alongside existing `profile` column.

## 4. Frontend Component (`BenefitsStep.tsx`, ~80 lines)

- Household size: number input (1-8)
- Monthly income: currency input ($0+)
- Enrolled programs: checkboxes (SNAP, TANF, Medicaid, ALL Kids, Childcare Subsidy, Section 8, LIHEAP, None)
- Dependents under 6: number input
- Dependents 6-17: number input
- "None" checkbox clears all others
- Skip button advances with defaults

## 5. Wizard Integration (`assess/page.tsx`)

Add `BENEFITS` step after `BARRIERS`, before `CREDIT`. Always shown. Always advanceable (all fields have defaults). Extract as separate component to avoid bloating assess/page.tsx (already at 417 lines).

## 6. Frontend Types

Add `BenefitsFormData` interface. Consider separate `benefits-types.ts` if `types.ts` (434 lines) is too large.
